Command Patternのメリット(利点)

* **要求を発行する側と実行する側を完全に分離できる(疎結合)**: これが最大のメリットです。リモコン(Invoker)は「ボタンを押すこと」に専念し、照明(Receiver)は「光ること」に専念します。お互いが相手の詳細を知る必要がないため、コードが非常にシンプルに保たれます(単一責任の原則：SRP)。
* **既存のコードに触れずに新機能を追加できる**: 新しい家電(例えばスマートロックなど)を追加したい場合、既存のリモコンのコードを1行も変更せずに、新しいコマンドの構造体を追加するだけで対応できます(開放・閉鎖の原則：OCP)。
* **「元に戻す(Undo)」機能が簡単に作れる**: コマンド自体が独立したオブジェクト(データ)になっているため、「さっき実行したコマンドを取り消す」というUndo機能を非常に美しく実装できます。書籍でもこのUndo機能が詳しく解説されています。
* **マクロやキューイング(予約実行)ができる**: 複数のコマンドをリスト(配列)に保存して「ボタン一つで全部実行する(マクロ)」や、コマンドをキューに溜めておいて「後で順番に実行する」といった高度な制御が容易になります。

Command Patternのデメリット(欠点・注意点)

* **構造体(クラス)の数が爆発的に増える**: 「照明をつける」「照明を消す」「テレビをつける」「テレビを消す」...といった具体的なアクションごとに、それぞれ新しく構造体(`XxxCommand`)を作る必要があります。結果としてファイルや小さな部品が大量に発生し、システム全体の構成を一目で把握するのが少し難しくなります。
* **単純な処理には「やりすぎ(Overkill)」になる**: リモコンのボタンが1つしかなく、今後も絶対に増えないような超シンプルなプログラムにこのパターンを適用すると、無駄にコードが長くなり、かえって読みにくくなります。Kent Beck氏の「まずは最もシンプルに作れ(Simple Design)」という教えに反する場合があります。
